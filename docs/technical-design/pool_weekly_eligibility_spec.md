# Pool Weekly Eligibility Spec

Updated: 2026-04-05

## Goal

กำหนดกติกา `pool` แบบใหม่ให้ชัดก่อนลงโค้ด โดยรอบนี้โฟกัสที่:
- แหล่งเงิน pool
- รอบคำนวณรายสัปดาห์
- เงื่อนไขสมาชิกที่มีสิทธิ์รับ
- หน้าต่างสิทธิ์แบบ 2 เดือน

ไฟล์นี้เป็น spec สำหรับ logic ใหม่ และควรใช้เป็นตัวอ้างอิงหลักก่อนแก้ implementation

## Core Rule Summary

- `pool fund` เท่ากับ `30%` ของ `PV รวมทั้งสัปดาห์`
- คำนวณและปิดรอบ `สัปดาห์ละ 1 ครั้ง`
- รอบปิดยอดคือ `ทุกวันอาทิตย์ 23:59`
- เงิน pool ของสัปดาห์นั้นถูกหารเท่ากันตาม `จำนวนสมาชิกที่มีสิทธิ์รับในรอบนั้น`

## Funding Rule

### Pool Source

- ใช้ `30%` ของ `PV รวม` ที่เข้ารอบในสัปดาห์นั้น
- สูตรหลัก:
  - `weeklyPoolFund = weeklyTotalPv * 0.30`

### Weekly Window

- 1 รอบคำนวณครอบคลุมยอด PV ของสัปดาห์นั้น
- ระบบต้องสรุปยอดและคำนวณเพียง `1 ครั้งต่อสัปดาห์`
- เวลา cutoff คือ `วันอาทิตย์ 23:59`

## Eligibility Rule

สมาชิกจะมีสิทธิ์รับ `pool` ได้ เมื่อผ่านทั้ง 2 เงื่อนไขนี้:

1. มี `แนะนำตรงอย่างน้อย 2 คน`
2. มี `B1 complete` ของตัวเองอย่างน้อย `1 ครั้งในเดือนนั้น`

## Direct Referral Qualification

- เงื่อนไข `แนะนำตรง 2 คน` เป็นเงื่อนไขสะสม
- ผ่านแล้วถือว่าผ่านต่อเนื่อง
- ไม่ต้องทำซ้ำทุกเดือน

สรุป:
- direct 2 คน คือ `base qualification`
- ถ้ายังไม่ผ่านข้อนี้ จะไม่ได้รับ pool แม้จะมี `B1 complete`

## B1 Qualification

- เงื่อนไขรายเดือนคือสมาชิกต้องมี `B1 complete` ของตัวเองอย่างน้อย `1 ครั้ง` ในเดือนนั้น
- การครบ `B1` ในที่นี้หมายถึง event การปิด `Board 1` ของสมาชิกคนนั้นเอง
- ไม่ใช่การที่ downline หรือ branch ใต้ตัวสมาชิกครบแทน

## Entitlement Window

เมื่อสมาชิก `ครบ B1 ครั้งแรกของเดือน`
- ให้เริ่มมีสิทธิ์รับ `pool` ตั้งแต่วันที่ครบ B1 นั้น
- สิทธิ์นี้คงอยู่ `จนจบเดือนถัดไป`

นิยามแบบสั้น:
- `1 B1 completion` เปิดสิทธิ์ให้ `2 เดือน`
- เดือนที่ครบ = เดือนเปิดสิทธิ์
- เดือนถัดไป = เดือนที่ยังต่อสิทธิ์ให้อัตโนมัติ

## Renewal Rule

ถ้าสมาชิกต้องการรับ `pool` ในเดือนที่ 3
- สมาชิกต้องมี `B1 complete` อีกอย่างน้อย `1 ครั้ง` ภายในเดือนที่ 2

สรุป:
- สิทธิ์ไม่ได้ถาวร
- ต้องมี `B1 complete อย่างน้อยเดือนละ 1 ครั้ง` เพื่อยืดสิทธิ์ไปเดือนถัดๆ ไป

## Weekly Payout Rule

ในรอบปิดทุกวันอาทิตย์ 23:59:

1. รวม `PV ทั้งสัปดาห์`
2. คำนวณ `pool fund = 30%` ของยอดนั้น
3. หาจำนวนสมาชิกที่มีสิทธิ์รับใน timestamp ปิดรอบนั้น
4. เอา `pool fund` หารเท่ากันตามจำนวนสมาชิกที่มีสิทธิ์

สูตรหลัก:
- `perEligibleMemberPayout = weeklyPoolFund / eligibleMemberCount`

ถ้าไม่มีสมาชิกที่ผ่านสิทธิ์:
- ยังต้องล็อกต่อว่าเงิน pool สัปดาห์นั้นจะ
  - ถูกพักไว้
  - ยกยอด
  - หรือทิ้งเป็น `0 payout`

รอบนี้ยังไม่ตัดสินใน spec นี้

## Qualification Timing Interpretation

เพื่อให้ behavior ชัดเจน ระบบควรตีความแบบนี้:

- ถ้าสมาชิกครบ `B1` ก่อนเวลา cutoff ของวันอาทิตย์
  - สมาชิกคนนั้นมีสิทธิ์เข้ารอบจ่ายของสัปดาห์นั้น
- ถ้าสมาชิกครบ `B1` หลัง cutoff
  - ให้เริ่มมีผลในรอบสัปดาห์ถัดไป

## Monthly Interpretation

คำว่า `เดือน` ใน spec นี้หมายถึง:
- เดือนปฏิทินปกติ
- เช่น `1 เมษายน - 30 เมษายน`

ตัวอย่าง:
- ถ้าครบ `B1` วันที่ `2026-04-10`
  - มีสิทธิ์รับตั้งแต่ `2026-04-10`
  - สิทธิ์ต่อเนื่องไปจน `2026-05-31 23:59:59`
- ถ้าต้องการมีสิทธิ์ต่อในเดือน `มิถุนายน 2026`
  - ต้องมี `B1 complete` อีกครั้งภายใน `พฤษภาคม 2026`

## Recommended Runtime Model

เพื่อให้ implementation ตรง spec ควรมีแนวคิดข้อมูลดังนี้:

- `weekly pool cycle`
  - startAt
  - endAt
  - totalPv
  - poolRate = `0.30`
  - poolFund
  - eligibleMemberCount
  - payoutPerMember

- `pool eligibility snapshot`
  - userId
  - hasTwoDirects
  - latestB1CompletionAt
  - qualifiedFrom
  - qualifiedUntil
  - qualifiedAtWeeklyClose

## Locked Decisions From This Round

สิ่งที่ถือว่าล็อกแล้วจากการคุยรอบนี้:

- pool ใช้ `30%` ของ `PV รวมรายสัปดาห์`
- ปิดรอบ `ทุกวันอาทิตย์ 23:59`
- direct อย่างน้อย `2 คน` เป็นเงื่อนไขตั้งต้น
- `B1 complete` เป็นเงื่อนไขรายเดือน
- `1 ครั้ง` ของ B1 เปิดสิทธิ์ให้ `เดือนปัจจุบัน + เดือนถัดไป`
- ถ้าจะรับต่อในเดือนถัดไปอีก ต้องมี `B1 complete` ภายในเดือนก่อนหน้า
- ใช้ timezone `Asia/Bangkok` เป็นมาตรฐานของรอบสัปดาห์และรอบเดือน
- ถ้าสัปดาห์นั้นไม่มีผู้มีสิทธิ์รับ ให้ `0 payout` และถือเป็น `company fallback` ของรอบนั้น
- ถ้าในเดือนเดียวกันมี `B1 complete` หลายครั้ง ให้ใช้ครั้งล่าสุดเป็นตัว refresh entitlement ภายในหน้าต่างสิทธิ์เดิม
- `weeklyTotalPv` นับจาก `approved normal orders` ที่ `pool visibility` เปิดใช้งานอยู่

## Implementation Notes

รอบ implementation แรกให้ใช้แนวทางนี้:

- ใช้ record `DailyPoolCycle` เดิมเป็น weekly close record ชั่วคราว
- ให้ `poolDate` หมายถึง `วันอาทิตย์ของรอบนั้น`
- ช่วงคำนวณคือ `วันจันทร์ 00:00:00` ถึง `วันอาทิตย์ 23:59:59.999` ตามเวลาไทย
- eligibility snapshot ยังคงเก็บในโครงเดิมได้ โดยตีความ `memberActive` เป็นสถานะ entitlement ของ pool window
